Type Erasure in C++: Teaching the Compiler to Forget — on Purpose

Or: How std::function stores a lambda, a function pointer, and a functor in the same box

You're building a transaction monitor for a SystemC simulation. The monitor needs to do several things every time a TLM transaction arrives: log it, measure its latency, run a protocol checker, collect coverage. Each of these is a separate object — a Logger, a LatencyTracker, a ProtocolChecker, a CoverageCollector. They have nothing in common except one thing: they all respond to a transaction.

You want to write this:

std::vector<TransactionHandler> handlers;
handlers.push_back(Logger{});
handlers.push_back(LatencyTracker{});
handlers.push_back(ProtocolChecker{});

for (auto& h : handlers)
    h.on_transaction(txn);

But the compiler won't let you. Logger, LatencyTracker, and ProtocolChecker are different types. A std::vector holds one type. You can't mix them.

This is the moment you need type erasure.

What Is Type Erasure?

Type erasure is a technique that lets you store objects of different concrete types behind a single uniform interface, without those types sharing a common base class, and without the caller knowing — or caring — what the real type is.

The "erasure" isn't deletion. It's concealment. The concrete type still exists, fully formed, somewhere in memory. But to the outside world, it has been replaced by a uniform wrapper that only exposes the behaviors you care about.

In one sentence: type erasure trades compile-time type knowledge for runtime flexibility, without exposing the mechanism to the caller.

The Root Problem: Why Templates Alone Aren't Enough

Templates are C++'s primary tool for abstraction without overhead. But they have a fundamental constraint: every template instantiation produces a different type.

template<typename T>
void call(T& handler, const Transaction& txn) {
    handler.on_transaction(txn);
}

// These are fine individually:
call(logger, txn);
call(tracker, txn);

// But this is impossible:
std::vector handlers;   // what type goes here?

A std::vector<Logger> holds only Logger objects. A std::vector<LatencyTracker> holds only LatencyTracker objects. There is no std::vector<AnyHandlerThatHasOnTransaction> — unless you build it.

Templates solve the single-call problem. Type erasure solves the heterogeneous-container problem.

The naive fix is inheritance:

struct HandlerBase {
    virtual void on_transaction(const Transaction&) = 0;
    virtual ~HandlerBase() = default;
};

std::vector<std::unique_ptr<HandlerBase>> handlers;

This works, but it forces every handler to inherit from HandlerBase. Your Logger from a third-party library? Doesn't inherit from HandlerBase. Your lambda? Can't inherit from anything. Type erasure solves this without requiring any change to the types being stored.

Three Mechanisms, One Idea

Before building something, it helps to see that type erasure already exists all around you in C++, in three flavors:

1. The C way — void pointers

The oldest form. qsort is the canonical example:

void qsort(void* base, size_t count, size_t size,
           int (*compare)(const void*, const void*));

The array's element type is erased into void*. The caller passes a comparator that knows the real type. The information is there — just split between the pointer and the function. Dangerous, but it works.

2. The OOP way — virtual dispatch

The classic approach: define an abstract interface, store pointers to the base. The vtable dispatches to the right implementation at runtime. Safe and clear, but requires inheritance.

3. The C++ way — template wrappers with virtual internals

This is the good stuff. You wrap any type T in a template-generated class that implements a virtual interface. The caller sees only the wrapper; the concrete type is hidden inside. This is how std::function, std::any, and most modern type-erasure patterns work.

Building It Yourself: A Step-by-Step Walkthrough

Understanding type erasure deeply means building a simplified std::function. Let's do exactly that: a MyFunction<void()> that can hold any callable with no arguments.

Step 1: Define the Concept

A concept (in the broad sense — what the wrapped type must support) is just the answer to: "what operations do I need from T?"

For a callable with no args, the concept is simple: - Must be callable as t() with no arguments - Must be copy-constructible (so we can copy our wrapper) - Must be destructible

That's it. No base class. No interface. Just a behavioral contract.

In C++20 you can make this explicit:

template<typename T>
concept Callable = requires(T t) {
    { t() } -> std::same_as<void>;
};

In C++17 and earlier, this contract lives in your head and in SFINAE or if constexpr. The concept is implicit but just as real.

Step 2: The Abstract Interface (Base)

The type-erased interface is a pure virtual base class. This is the shape that the outside world sees:

struct CallableBase {
    virtual void call() = 0;
    virtual CallableBase* clone() const = 0;
    virtual ~CallableBase() = default;
};

This is the "erased" view. It knows nothing about the concrete type. It only knows: you can call this, and you can copy it.

Step 3: The Template Wrapper — Where Concrete Types Are Born

This is the crucial piece. CallableModel<T> is a template that: - Stores a specific concrete type T - Implements the virtual interface by delegating to T

template<typename T>
struct CallableModel : CallableBase {
    T callable_;

    explicit CallableModel(T t) : callable_(std::move(t)) {}

    void call() override {
        callable_();   // delegate to the real type
    }

    CallableBase* clone() const override {
        return new CallableModel<T>(callable_);
    }
};

This is where the compiler instantiates a concrete type. When you write:

MyFunction f = [](){ std::cout << "hello\n"; };

The compiler sees T = lambda_type_XYZ. It stamps out a new class CallableModel<lambda_type_XYZ>, complete with: - A member of type lambda_type_XYZ - An overridden call() that invokes lambda_type_XYZ::operator()() - An overridden clone() that copies the lambda

This stamping-out happens at the point of assignment or construction — in the template that creates the CallableModel. The key insight: the concrete type is captured forever inside CallableModel<T>, and nothing outside ever needs to know T again.

Step 4: The Owning Class — The Public API

The owning class is what users actually hold. It manages the CallableBase* and provides the clean interface:

class MyFunction {
    CallableBase* ptr_ = nullptr;
public:
    MyFunction() = default;

    // This template constructor is the "type erasure point"
    template<typename T>
    MyFunction(T callable)
        : ptr_(new CallableModel<T>(std::move(callable))) {}

    // Copy: use clone() — works without knowing T
    MyFunction(const MyFunction& other)
        : ptr_(other.ptr_ ? other.ptr_->clone() : nullptr) {}

    // Move
    MyFunction(MyFunction&& other) noexcept
        : ptr_(std::exchange(other.ptr_, nullptr)) {}

    MyFunction& operator=(MyFunction other) noexcept {
        std::swap(ptr_, other.ptr_); return *this;
    }

    // The erased call — no template, no T, just behavior
    void operator()() const {
        if (ptr_) ptr_->call();
    }

    ~MyFunction() { delete ptr_; }
};

Now you can write:

MyFunction f1 = []{ std::cout << "lambda\n"; };
MyFunction f2 = &some_free_function;

struct Functor { void operator()() { std::cout << "functor\n"; } };
MyFunction f3 = Functor{};

std::vector<MyFunction> tasks = {f1, f2, f3};
for (auto& t : tasks) t();  // works. all different types. one interface.

The vector holds MyFunction objects. Each MyFunction holds a CallableBase*. Each CallableBase* points to a different CallableModel<T>. The concrete types — lambda, function pointer, functor — are hidden inside. Erased.

Step 5: Small Buffer Optimization (SBO)

The implementation above heap-allocates every callable. For small callables (like a lambda capturing nothing, or a function pointer), this is wasteful. Real implementations of std::function use a Small Buffer Optimization: a small stack-allocated buffer inside the owning class. If the callable fits, no heap allocation happens.

class MyFunction {
    static constexpr size_t SBO_SIZE = 32;
    alignas(std::max_align_t) char buf_[SBO_SIZE];
    CallableBase* ptr_ = nullptr;
    bool heap_ = false;

    template<typename T>
    void store(T callable) {
        if (sizeof(CallableModel<T>) <= SBO_SIZE) {
            ptr_ = new (buf_) CallableModel<T>(std::move(callable));
            heap_ = false;
        } else {
            ptr_ = new CallableModel<T>(std::move(callable));
            heap_ = true;
        }
    }
public:
    // … rest of interface
    ~MyFunction() {
        if (ptr_) {
            ptr_->~CallableBase();          // always call destructor
            if (heap_) ::operator delete(ptr_);  // only free if heap
        }
    }
};

This is why small lambdas through std::function are nearly free, while lambdas capturing large state trigger a heap allocation.

How the Compiler Handles It: The Concrete Type Timeline

This is the mental model that makes type erasure click:

User code:
    MyFunction f = some_lambda;  // ← compiler sees T = lambda_type

Compiler action at this line:
    1. Deduces T = lambda_type from the constructor argument
    2. Instantiates CallableModel<lambda_type> — generates a brand new class
    3. Constructs a CallableModel<lambda_type> in memory (stack or heap)
    4. Stores a CallableBase* pointing to it inside MyFunction

After this line:
    - Nobody outside MyFunction knows T anymore
    - The concrete type is "forgotten" by the type system
    - Only the vtable pointer inside the CallableBase knows how to call it
    - You can pass MyFunction anywhere — copy it, move it, put it in a vector — T is irrelevant

When f() is called:
    - ptr_->call() → vtable lookup → CallableModel<lambda_type>::call()
    - → lambda_type::operator()()
    - The compiler generated this dispatch path at template instantiation time

The concrete type lives on as a fully real, fully typed object in memory. The type erasure only happens at the interface boundary — the CallableBase* level. Below that, everything is fully typed and can be inlined by the compiler within CallableModel<T>::call().

The C++ Standard Library: Type Erasure Everywhere

Once you know what to look for, you see it everywhere in <functional>, <memory>, and <any>.

std::function<R(Args...)>

The most famous example. Stores any callable matching a signature. Uses SBO internally (typically 16–32 bytes). The template parameter R(Args...) is the concept — the only thing you need to know about T.

std::function<void(int)> f;
f = [](int x){ std::cout << x; };    // lambda — type erased
f = &printf_style_fn;                  // function pointer — type erased
f = std::bind(&Obj::method, &obj, _1); // bound member — type erased

std::any

Type erasure without any behavioral concept. Stores any copy-constructible type:

std::any a = 42;
a = std::string("hello");
a = SomeComplexStruct{};

// Type is recoverable via std::any_cast, but the container is typeless
int x = std::any_cast<int>(a);  // throws if wrong type

Internally: same pattern — virtual base, template model, SBO. The "concept" is just "copy-constructible and destructible."

std::variant<T1, T2, T3>

A closed form of type erasure — the set of possible types is fixed at compile time. std::variant<int, float, std::string> can hold exactly those three types, uses a discriminant tag instead of vtable, and enables std::visit dispatch:

std::variant<int, float, std::string> v = 42;
std::visit([](auto& val){ std::cout << val; }, v);

Prefer variant when the set of types is known at compile time — it's typically faster (no heap, no vtable, potentially constexpr) and more explicit about what types are allowed.

std::shared_ptr<T> — The Deleter

This is a subtle one. std::shared_ptr<T> can be constructed with a custom deleter of any type:

auto p1 = std::shared_ptr<FILE>(fopen("x", "r"), fclose);
auto p2 = std::shared_ptr<void>(malloc(64), free);
auto p3 = std::shared_ptr<Widget>(new Widget, [](Widget* w){ w->cleanup(); delete w; });

All three are std::shared_ptr<FILE>, std::shared_ptr<void>, std::shared_ptr<Widget> — the deleter type is erased from the shared_ptr type. It lives in the control block as a type-erased callable. This is why shared_ptr copying works without knowing the deleter type.

std::packaged_task<R(Args...)>

Like std::function, but the callable is movable-only and the result is stored in a std::future. Type-erases the callable for use with thread pools.

std::type_index

A lightweight, copyable wrapper over std::type_info. Type-erases the type itself into a hashable, comparable value. Used to build type-keyed maps at runtime:

std::unordered_map<std::type_index, std::any> registry;
registry[typeid(Widget)] = WidgetFactory{};

C++20 std::stop_token / std::stop_callback

std::stop_callback type-erases the callback registered for cooperative cancellation, so the std::stop_token doesn't need to know the callback's type.

Ranges and std::function_ref (C++26 preview)

std::ranges::views use type-erased adaptors under the hood in some contexts. std::function_ref (proposed for C++26) is a non-owning, zero-overhead type-erased callable reference — no heap allocation, no SBO needed.

Use Cases in Depth

1. Task Queues and Thread Pools

The canonical production use case. A thread pool needs a queue of work items with different types:

class ThreadPool {
    std::queue<std::function<void()>> tasks_;
    std::mutex mtx_;
public:
    template<typename F>
    void enqueue(F&& task) {
        std::lock_guard lock(mtx_);
        tasks_.push(std::function<void()>(std::forward<F>(task)));
    }

    void run_next() {
        std::function<void()> task;
        {
            std::lock_guard lock(mtx_);
            if (tasks_.empty()) return;
            task = std::move(tasks_.front());
            tasks_.pop();
        }
        task();  // concrete type was erased; we just call it
    }
};

// Usage
pool.enqueue([&]{ process_packet(pkt); });
pool.enqueue(std::bind(&Sim::run_step, &sim, clock));
pool.enqueue(&flush_log);

The thread pool doesn't know — and shouldn't care — what the tasks are. std::function<void()> is the type erasure layer.

Insight for the seasoned programmer: For high-throughput thread pools, std::function overhead (potential heap alloc per enqueue) matters. Consider a custom type-erased callable with a larger SBO, or the move_only_function from C++23, which avoids the requirement to be copy-constructible and thus enables more aggressive move semantics.

2. Event Systems and Signal/Slot

A type-erased event system:

template<typename... Args>
class Signal {
    std::vector<std::function<void(Args...)>> slots_;
public:
    template<typename F>
    void connect(F&& f) {
        slots_.emplace_back(std::forward<F>(f));
    }

    void emit(Args… args) {
        for (auto& slot : slots_)
            slot(args…);
    }
};

// Usage
Signal<const Transaction&> on_txn;

on_txn.connect([&](const Transaction& t){ logger.log(t); });
on_txn.connect([&](const Transaction& t){ tracker.record(t); });
on_txn.connect(&protocol_check);

on_txn.emit(txn);  // all three fire

This solves exactly the SystemC monitor problem from the opening story. Different handler types, one signal, zero coupling.

3. SystemC TLM Extension Mechanism

The TLM 2.0 standard uses type erasure directly. tlm_generic_payload supports user-defined extensions via tlm_extension<T>:

// The type-erased base
class tlm_extension_base {
public:
    virtual tlm_extension_base* clone() const = 0;
    virtual void free() { delete this; }
    virtual ~tlm_extension_base() = default;
};

// The template wrapper — this IS the type erasure model
template<typename T>
class tlm_extension : public tlm_extension_base {
public:
    static unsigned int ID;  // unique ID per concrete T
    // …
};

// User-defined extension
struct LatencyExtension : tlm::tlm_extension<LatencyExtension> {
    sc_core::sc_time start_time;
    tlm_extension_base* clone() const override {
        return new LatencyExtension(*this);
    }
    // …
};

// Payload stores extension via type-erased pointer
tlm::tlm_generic_payload txn;
txn.set_extension(new LatencyExtension{sc_time_stamp()});

// Later, anywhere in the model:
LatencyExtension* ext = nullptr;
txn.get_extension(ext);  // type-safe retrieval
if (ext) std::cout << "start: " << ext->start_time << "\n";

The payload holds a tlm_extension_base* array — it knows nothing about the concrete extension types. The concrete type is recovered via tlm_extension<T>::ID — a runtime type tag generated once per concrete T. This is textbook type erasure.

4. Heterogeneous Property Maps

A type-safe property map where values can be any type:

class PropertyMap {
    std::unordered_map<std::string, std::any> props_;
public:
    template<typename T>
    void set(const std::string& key, T value) {
        props_[key] = std::move(value);
    }

    template<typename T>
    T get(const std::string& key) const {
        return std::any_cast<T>(props_.at(key));
    }
};

PropertyMap cfg;
cfg.set("timeout_ns", 100);
cfg.set("name", std::string("initiator"));
cfg.set("enable_coverage", true);

int t = cfg.get<int>("timeout_ns");

This pattern is invaluable for configuration systems, test harnesses, and generic component interfaces where the set of properties isn't known at compile time.

5. Policy Injection Without Template Proliferation

One of the most practical — and underappreciated — uses of type erasure in production C++. Templates are excellent for zero-overhead abstraction, but they have a cost: template parameters propagate through the call stack. If your Cache<T, AllocPolicy, EvictionPolicy> has seven template parameters, every class that uses it gets those parameters too.

Type erasure lets you stop the bleeding:

// Without type erasure: template parameters propagate upward
template<typename EvictionPolicy>
class Cache {
    EvictionPolicy policy_;
    // ...
};
template<typename E>
class Simulator { Cache<E> cache_; /* ... */ };
template<typename E>
class TopLevel { Simulator<E> sim_; /* ... */ };
// Now TopLevel<LRU>, TopLevel<LFU>, TopLevel<Random>...
// Every layer knows about EvictionPolicy. Compile times suffer.

// With type erasure: stop at the interface boundary
class Cache {
    std::function<void(CacheKey)> on_evict_;   // type-erased policy
public:
    template<typename EvictionPolicy>
    Cache(EvictionPolicy p) : on_evict_(std::move(p)) {}
};

class Simulator { Cache cache_; /* no template params */ };
class TopLevel  { Simulator sim_; /* no template params */ };
// Concrete policy chosen once at construction. Nobody else cares.

This is one of the best arguments for std::function over pure templates in class member positions — it contains the template blast radius.

6. Plugin Systems and Dynamic Loading

When loading plugins at runtime (dlopen/LoadLibrary), the plugin's types can never be known at compile time. Type erasure is mandatory:

// Plugin interface — the erased concept
struct IPlugin {
    virtual void execute(Context&) = 0;
    virtual std::string name() const = 0;
    virtual ~IPlugin() = default;
};

// Plugin factory — returned from dlsym
extern "C" IPlugin* create_plugin();

// Host loads plugin without knowing its type
void* lib = dlopen("my_plugin.so", RTLD_NOW);
auto factory = reinterpret_cast<IPlugin*(*)()>(dlsym(lib, "create_plugin"));
std::unique_ptr<IPlugin> plugin(factory());

plugin->execute(ctx);  // concrete type? completely unknown. completely irrelevant.

Gotchas: Where Type Erasure Hurts

1. Heap Allocation by Default

Every std::function that doesn't fit in the SBO buffer allocates. In tight loops, this is measurable. Profile before assuming it's fine.

// Likely SBO (small lambda, no capture) — fast
std::function<void()> f = []{ ++counter; };

// Likely heap (captures a vector or a large struct) — heap alloc
std::vector<int> big_data(1000);
std::function<void()> g = [big_data]{ process(big_data); };

2. You Lose the Type at the Boundary — Forever

Once erased, the type can only be recovered via std::any_cast, dynamic_cast, or a type-tag mechanism. If you need to recover the type often, that's a design smell — you may want std::variant instead.

3. std::function Is Not Zero-Cost

A direct template call is inlinable. A std::function call goes through a vtable (or function pointer internally). The call itself isn't free. For functions called millions of times per second, this matters.

// Template: inlinable, zero overhead
template<typename F>
void applyN(F f, int n) { for (int i = 0; i < n; ++i) f(i); }

// std::function: virtual dispatch per call
void applyN(std::function<void(int)> f, int n) { for (int i = 0; i < n; ++i) f(i); }

4. std::function Requires Copyability

std::function requires the wrapped type to be copy-constructible. Unique lambdas that capture unique_ptr can't go into std::function. Use std::move_only_function (C++23) or a hand-rolled move-only wrapper.

auto p = std::make_unique<Resource>();
// This fails — can't copy a unique_ptr:
std::function<void()> f = [p = std::move(p)]{ p->use(); };  // ERROR

// C++23 solution:
std::move_only_function<void()> f = [p = std::move(p)]{ p->use(); };  // OK

5. Object Slicing (the OOP variant)

If you use raw pointers or references to a base class (the OOP style of type erasure), forgetting virtual on the destructor causes slicing and leaks. Always use virtual ~Base() = default; on any type-erased base.

6. Overhead of std::any

std::any is the most permissive form of type erasure — it stores anything. It's also the heaviest. Avoid it in hot paths or tight containers. Prefer std::variant when the type set is known.

When to Use Type Erasure vs Templates

Situation Prefer Templates Prefer Type Erasure

Zero-overhead is critical ✓
Called in a hot loop (>1M/s) ✓
Single call site, no container ✓
Types not known at compile time ✓
Storing mixed types in a container ✓

The practical rule of thumb: use templates at the point of use; use type erasure at storage boundaries.

Compiler and Performance Notes

Where Concrete Types Are Instantiated

The compiler generates a new CallableModel<T> specialization at the point where MyFunction(T t) (or std::function<F>(T t)) is called — the type-erasure point. This is typically in a .cpp file where the concrete T is known. Everything after that point sees only CallableBase* or std::function<F> — the concrete type is invisible.

Compile time implication: Type erasure reduces compile time compared to full template propagation. The concrete instantiation happens once, in one translation unit. Template-heavy code makes every including TU re-instantiate everything.

The Vtable Dispatch Cost

A virtual function call typically costs 1–5 ns on modern hardware — a table lookup plus an indirect branch. Branch predictors are good at predicting virtual calls that always go to the same concrete type. If your type-erased object always holds a Logger, the branch predictor will learn this quickly and the indirect call becomes nearly free.

But if the concrete type changes unpredictably across calls — the branch predictor can't help, and you'll pay the full misprediction cost. In that case, consider std::variant + std::visit, which the compiler can turn into a branch table.

SBO Sizes in Practice

GCC's libstdc++ and Clang's libc++ use different SBO sizes for std::function: - libstdc++: ~16 bytes usable (after overhead) - libc++: ~24 bytes usable

A lambda capturing a single pointer fits. A lambda capturing a std::string may not. Measure with -O2 -S and check whether operator new appears in the output for your specific callable.

std::any and the Copy Overhead

std::any copies on every copy. If your std::any holds a large struct, every copy of the std::any deep-copies that struct. Use std::reference_wrapper or std::shared_ptr inside std::any to share ownership if deep-copy is expensive.

The Bigger Picture

Type erasure sits at the intersection of two deep ideas in C++: zero-cost abstraction (templates) and runtime polymorphism (virtual dispatch). Neither alone is sufficient for the problems type erasure solves. Type erasure combines them: the concrete type gets a full template instantiation with all the zero-cost benefits — inlining, no dynamic dispatch within the model — while the outside world gets a stable, uniform, runtime-polymorphic interface.

The compiler doesn't forget the type. You ask it to. And it does so precisely and safely, generating exactly the dispatch machinery needed for the type you gave it, at the exact moment you gave it, in the exact translation unit where that type is visible.

That's not forgetting. That's encapsulation at its most deliberate.
